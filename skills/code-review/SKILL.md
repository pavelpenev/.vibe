# Code Review

Review code changes for correctness, security, bugs, typos, and style issues. Compares code against the described change intent and identifies problems.

## Triggering Conditions

Activates when user requests code review:
- `/code-review` - review uncommitted changes
- `/code-review <file>` - review specific file(s)
- `/code-review <branch>` - review branch diff against main
- `/code-review PR <number>` - review pull request changes
- "review my changes"
- "review this code"
- "code review"

## Step 1: Parse Request

Extract:
- **Target**: Files, diff, branch, or PR to review
- **Description**: What the change should accomplish
- **Scope**: Narrow (default) vs deep review

If target is unspecified:
- Check for uncommitted changes via `git status`
- If changes exist: use `git diff` for unstaged, `git diff --staged` for staged
- If no changes: prompt "No uncommitted changes found. Specify a file or branch to review."

If description is missing:
- **Prompt user**: "What should these changes accomplish? (e.g., 'add user authentication', 'fix login bug')"
- Wait for user response before proceeding

## Step 2: Determine Scope and Get Content

Based on target:

**Uncommitted changes (no target):**
```bash
git diff --name-only  # unstaged files
git diff --staged --name-only  # staged files
```
Review all changed files. For each file, get the full content (not just diff) for complete analysis.

**Specific file(s):**
Read the complete content of each specified file.

**Branch:**
```bash
git diff main...<branch> --name-only  # get changed files
git diff main...<branch>  # get full diff
```
Review all files changed in the branch diff. Read full file content for each.

**PR number:**
```bash
git fetch origin pull/<number>/head:<pr-branch>
git diff main...<pr-branch> --name-only
```
Then proceed as branch review.

**Skip:**
- Binary files (detect via `file` command or null bytes)
- Auto-generated files (detect common patterns: "DO NOT EDIT", "Generated by", "Auto-generated", minified JS, license header only)

## Error Handling

**No changes found via `git status`**:
- Check if target mentions branch comparison (e.g., "feature vs main", "compare branch1 and branch2")
- If so: Use `git diff branch1..branch2 --name-only` to get changed files
- If target mentions specific commit range: Use `git diff commit1..commit2 --name-only`
- If none apply: Prompt "No changes found to review. Specify a file, branch, or commit."

**File access issues**:
- If a specified file doesn't exist: Report as error, skip that file
- If file is unreadable: Report with reason, skip that file
- If git command fails (not a repo): Report and fall back to direct file paths from target

**Commit messages**:
- For commit-based reviews, use `git show <ref> --no-patch --format=%B` to capture commit message
- Use commit message as the description for Correctness Check if user description is missing

## Step 3: Run Checks

Run all checks on each file. All checks are language-agnostic and work on raw text.

### Check Categories

**1. Correctness (Matches Intent)**
Compare code against the description using semantic understanding:
- Does the code implement what the description says?
- Are there obvious mismatches between description and implementation?
- Flag if description is too vague to verify ("fix stuff", "update code")

**2. Security**
- Hardcoded secrets: API keys, passwords, tokens, private keys
  - Patterns: `api[_-]?key`, `password`, `secret`, `token`, `private[_-]?key`, `access[_-]?key`, `auth`, `credential`
  - Entropy check: strings with high entropy (3.5+ bits/char) that look like secrets
  - Quote patterns: `"..."`, `'...'` containing suspicious keywords
- **Blocking**: Any hardcoded secret finding = BLOCKING ISSUE

**3. Bugs**
- Variables used before definition (simple tracking within scope)
- Variables defined but never used
- Functions that should return a value but don't (detect by checking if function has any return statements and ends without one)
- Exception handlers that catch all exceptions without handling (`except:` or `catch (`)
- Missing error handling for operations that can fail (file I/O, network calls)

**4. Typos**
- Variables used only once in a scope (possible typo)
- Variables with names similar to others (edit distance <= 2): `user_nme` vs `user_name`, `countr` vs `country`
- Common misspellings in identifiers: `recieve`, `seperate`, `definately`, `occured`
- Common misspellings in comments/strings: `tough`, `through`, `their`, `there`

**5. Style**
- Line length > 120 characters
- Consecutive blank lines (>2)
- Trailing whitespace
- Mixed tabs and spaces
- File missing trailing newline

**6. Documentation**
- Missing docstrings on public functions/classes
- Missing comments for complex logic
- Outdated comments that don't match code
- TODO/FIXME comments without issue references
- Missing type hints (where applicable)

**Note on Test Files**: Apply all checks to test files, but be more lenient on style (e.g., allow longer lines for test data). For test files, prioritize: correctness > coverage > style.

### Severity Levels

- **ERROR** (Blocking): Security issues (hardcoded secrets only)
- **WARNING**: Bugs, typos, significant style violations (>5 per file)
- **INFO**: Style issues (<=5 per file), minor suggestions

## Step 4: Generate Report

Format as markdown:

```markdown
## Code Review Report

**Target:** {files/diff/branch reviewed}
**Description:** {user-provided description}
**Status:** {PASSED/FAILED}

---

### Summary

| Category | Errors | Warnings | Info |
|----------|--------|----------|------|
| Correctness | X | - | - |
| Security | X | - | - |
| Bugs | - | X | - |
| Typos | - | X | - |
| Style | - | - | X |
| Documentation | - | X | - |

---

### Blocking Issues (Must Fix)

{If any ERROR findings, list them here. Empty section = no blockers.}

- **[SECURITY]** Hardcoded {type} detected at `{file}:{line}`
  ```
  {code snippet}
  ```
  **Fix:** Move to environment variable or secret manager

---

### Warnings (Should Fix)

{List all WARNING findings}

- **[BUGS]** Variable `{var}` used before definition at `{file}:{line}`
- **[BUGS]** Variable `{var}` defined but never used at `{file}:{line}`
- **[TYPOS]** `{var1}` is similar to `{var2}` (edit distance: N) at `{file}:{line}`
- **[TYPOS]** Possible typo `{word}` (did you mean `{suggestion}`?) at `{file}:{line}`
- **[STYLE]** File has {N} style warnings (>5 threshold) at `{file}`

---

### Info (Consider Fixing)

{List all INFO findings}

- **[STYLE]** Line {N} exceeds 120 characters at `{file}`
- **[STYLE]** Consecutive blank lines at `{file}:{line}`
- **[STYLE]** Trailing whitespace at `{file}:{line}`
- **[DOCUMENTATION]** Missing docstring on {function/class} at `{file}:{line}`
- **[DOCUMENTATION]** Missing type hints on {function} at `{file}:{line}`

---

### Correctness Check

**Description:** {user description}

**Assessment:** {PASS/FAIL/WARNING}

{If PASS: Code appears to implement the described functionality}
{If FAIL: Code does not match description - {specific discrepancies}}
{If WARNING: Description is vague, cannot fully verify - {suggestions for better description}}

---

**Next Steps:**
- {Fix N blocking issues before committing/merging}
- {Address M warnings to improve code quality}
- {Consider K info suggestions for optimization}
```

## Step 5: Final Status

- If any ERROR findings: **Status = FAILED** (blocking)
- If no ERROR but has WARNING: **Status = PASSED WITH WARNINGS**
- If only INFO or no findings: **Status = PASSED**

State clearly: "This review found BLOCKING issues that must be fixed before merging." or "No blocking issues found."

## Configuration

Users can configure via AGENTS.md:

```markdown
## Code Review Settings
- Max line length: 120 (default)
- Style warning threshold: 5 (default)
- Block on secrets: true (default)
```

## Anti-Rationalization

| User concern | Response |
|--------------|----------|
| "This is too slow" | Review is fast for typical changes. Large files get warnings, not blocked. |
| "Too many false positives" | We start conservative. Report issues and we'll improve. |
| "I don't need this" | Even small changes benefit from review. Takes seconds. |
| "Just auto-fix" | Some issues require human judgment. We identify, you decide. |

## Red Flags

- User tries to skip review for changes with secrets
- User ignores blocking issues
- User provides extremely vague descriptions repeatedly
- Review takes >10 seconds (report as potential issue)
